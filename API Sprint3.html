Projetaí

Resumo do Backend - API Sprint 3

Uma Jornada do Zero a uma API REST Completa

O Desafio Inicial

  • Objetivo: Reconstruir o "Projetaí", evoluindo de uma aplicação simples para uma solução robusta.
  • Problema: A versão original era "client-only", com dados salvos localmente (localStorage), limitando escalabilidade e compartilhamento.
  • Solução: Criar uma arquitetura Cliente-Servidor com um backend dedicado, banco de dados relacional e uma API REST profissional.

A "Stack" Tecnológica

  • Linguagem: Java 17 (LTS)
  • Framework: Spring Boot
  • Acesso a Dados: Spring Data JPA & Hibernate
  • Banco de Dados: MySQL
  • Build & Dependências: Maven
  • Documentação: OpenAPI 3 / Swagger (com springdoc)
  • Testes: Postman

Arquitetura da Solução

  • Controller Layer (@RestController): A "porta de entrada" da API. Recebe requisições HTTP e delega para a camada de serviço.
  • Service Layer (@Service): O "cérebro" da aplicação. Contém toda a lógica de negócio e gerencia as transações (@Transactional).
  • Repository Layer (@Repository): A camada de acesso aos dados. Abstrai a comunicação com o banco através de interfaces que estendem JpaRepository.
  • Model Layer (@Entity): A "planta baixa" dos dados. Classes Java que espelham as tabelas do banco.

Modelo de Dados (Entidades)

  • Project: O contêiner principal. Possui id, name, description, dueDate e um code único.
  • Task: Pertence a um Project. Possui id, title, description, dueDate, status e priority.
  • ChecklistItem: Pertence a uma Task. Possui id, text e checked.
  • Comment: Pertence a uma Task. Possui id, text, authorName e createdAt.
  • Relacionamentos: Project 1 -> N Tasks, Task 1 -> N ChecklistItems, Task 1 -> N Comments.

Desafios e Aprendizados

  • Configuração de Ambiente: A importância de usar versões Java LTS estáveis (o bug do JDK 25).
  • Persistência e Transações: Depuração profunda do `UPDATE` que não acontecia, explorando @Transactional, save() vs. saveAndFlush().
  • Contrato da API: Correção de bugs de desserialização JSON devido a nomes de propriedades diferentes no DTO (checked vs isChecked).
  • Relacionamentos Bidirecionais: Solução da recursão infinita no JSON com @JsonManagedReference e @JsonBackReference.
  • Build do Projeto: Depuração do build Maven, corrigindo encoding e a configuração do plugin springdoc para gerar o openapi.json estático.
